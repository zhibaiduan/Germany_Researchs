公司全景：从五个人到一座城池

SAP 的历史沿革、战略演变与今天的业务逻辑

2026-05-18 18 分钟阅读 历史 战略 商业模式

一个朴素的问题

想象你是一家制造企业的老板。你的公司有 5,000 名员工，分布在三个国家，每天产生几千张采购订单、几百笔财务分录、几十批库存进出。你的财务总监问你："我们这个月的利润是多少？"你的采购经理问你："这批原料什么时候到？"你的德国工厂长问你："中国分公司的库存还能撑几天？"

在没有统一系统的情况下，你给不了答案。财务用一套软件，采购用另一套，仓库用 Excel。月底大家花两周对账，然后告诉你一个"大概"的数字。

SAP 存在的意义，就是让这种混乱变得有序。

这不是一个关于"好用"的故事。这是一个关于"能跑起来"的故事。SAP 之所以成为全球最大的企业软件公司，不是因为它做出来的软件最好看、最聪明，而是因为它解决了这个最朴素的问题：让一家企业的各个部分在同一个系统里运转。

第一章：它解决什么问题、靠什么赚钱

它卖的是什么

SAP 的核心产品叫 ERP（企业资源计划）系统。说白了就是一套软件，把一家公司的财务、采购、库存、生产、销售、人事全部装进一个系统里。

想象一座城市的运转需要供水、供电、交通、通信各自协调，ERP 就是一家企业的"城市操作系统"。没有它，每个部门各自为政，有了它，至少大家在同一个底图上说话。

SAP 的产品线不止 ERP。它还有：

  • SuccessFactors：管人事的（招聘、绩效、薪酬）
  • Ariba：管采购的（供应商管理、招标）
  • Concur：管差旅报销的
  • BTP（Business Technology Platform）：管"在上面做开发"的平台

但 ERP 是核心，是根基。其他产品都是围绕它长出来的。

它的生意怎么做的

SAP 卖软件的方式经历了两次变化。

一开始是"买断制"。 客户一次性付一大笔钱买许可证（比如 €5M），然后每年再付 22% 的维护费（比如 €1.1M/年），换来的好处是：可以一直用这个软件，不用再付授权费。这套模式让 SAP 在 1990 年代和 2000 年代赚得盆满，大企业买一套 SAP，相当于给自己买了一套用几十年的基础设施。

类比：买断制

你付一大笔首付买下房子（一次性许可证费），然后每年交物业费（维护费）。房子是你的，你可以在里面装修、改造、住一辈子。

现在正在变成"订阅制"。 客户不再一次性买许可证，而是每年付一笔订阅费（RISE with SAP），软件、基础设施、支持全包在里面。一年不付，系统就停用。

类比：订阅制

你按月交房租（订阅费），包物业包水电。房子不是你的，但你不用操心维修。房东（SAP）可以随时给你换新家具（升级版本）。

从客户视角看，订阅制的好处是不用一次性掏一大笔钱，坏处是长期算下来付的钱更多。 SAP 为什么强行推订阅制？不是因为它对客户更好，而是对 SAP 自己更好，订阅收入比一次性收入更可预测、更稳定，资本市场愿意给 SAP 更高的估值倍数。

客户为什么付钱

不是因为 SAP 的产品最好用。事实上，很多用户私下告诉你 SAP 的界面"像 1990 年代的东西"。

客户付钱的理由非常实在：走不了。

SAP 做对了三件事，可能不是设计出来的，而是历史形成的：

第一，你的业务已经被它的代码绑住了。 一家用了 15 年 SAP 的企业，用 ABAP 语言写了成千上万条定制化程序来适配自己的独特流程：怎么算库存、怎么审批采购、怎么做成本核算。换掉 SAP，这些程序全部报废。

第二，你的数据被锁在它的系统里。 十年的财务记录、供应商清单、客户交易历史、审计轨迹都在 SAP 的数据库中。这些数据不是简单"导出 Excel"就能迁移到另一个系统的，数据模型不一样、业务规则不一样、合规要求不一样。你试过把一家大公司的财务数据从一套系统搬到另一套吗？那不是一个工程师周末能干完的活，而是一个一年以上的项目。

第三，你的整个生态都是它的人。 帮你实施 SAP 的咨询公司（Accenture、Deloitte）、在你系统上做插件的独立软件商、甚至你招聘的 SAP 顾问，整个行业都围绕着 SAP 运转。换掉 SAP 不只是换一个软件供应商，而是要换掉整个支持体系。

客户逻辑

我对你不满，但我走不了。我们的关系不是"你做得好所以我买"，而是"你已经长在我身上了，我只能继续付钱"。

第二章：54 年，三个阶段的故事

第一个故事：五个人的赌注（1972-1992）

1972 年，五位在 IBM 德国工作的工程师辞职了。在当时的德国，从 IBM 辞职是一件很不靠谱的事，IBM 是当时最稳定的雇主。他们为什么走？

他们的想法非常具体：企业数据不应该等到月底才能看。

当时企业的运作方式是：业务部门每天填纸质单据，财务部月底统一录入大型机，几天后出报表。如果数字对不上，再花几周人工核对。这五位工程师（Dietmar Hopp、Hans-Werner Hector、Hasso Plattner、Klaus Tschira 和 Claus Wellenreuther）认为这个流程是浪费时间的。为什么不从一开始就让数据实时录入、实时计算？

这个想法被 IBM 拒绝了，IBM 认为"实时计算"不是企业客户需要的功能。所以他们自己出来干。

第一批客户来得很慢。第一个客户是德国的化工公司。前四年他们只做了十几家客户。直到 1980 年代末，他们的产品 R/2 才开始在德国制造业中普及。

这个阶段的 SAP，产品思维非常德国：功能追求完整和准确，界面好不好看是次要的。 这个基因一直留到了今天。

第二个故事：从德国走向世界（1992-2010）

1992 年，SAP 推出了 R/3。这是它历史上最关键的一次产品发布。

R/3 做了一件简单但革命性的事：把企业软件从大型机搬到客户机/服务器架构上。 这意味着企业不再需要买 IBM 的大型机，用普通服务器就能跑 SAP。这意味着 SAP 不再是 IBM 硬件生态的一部分，而是可以跑在任何一家公司的服务器上。

这个决定的回报超出了所有人的预期。

在整个 1990 年代，SAP 从一家德国公司变成了一家全球公司。员工从几千人涨到两万多人，客户从德国制造业扩展到全行业、全球各地。R/3 不仅让 SAP 赚到了钱，还让它的产品成为"企业 ERP 的标准"，你做 ERP 选型时，如果不把 SAP 列进候选清单，董事会会觉得你不专业。

但 R/3 的成功也埋下了一个隐患。

R/3 时代有一个模式：客户买标准产品，然后请咨询公司用 ABAP（SAP 自有的编程语言）做大量定制开发。每家企业都按自己的业务流程改了 SAP。这样做的结果是：SAP 的销售很顺利（客户觉得"任何需求都能定制"），但客户的系统变得越来越复杂、越来越难升级。每个客户的 SAP 系统都变得独一无二。 这就意味着，以后 SAP 想让他们升级到新版本，工作量很大。

第三个故事：两次转型，一次还没成功（2010-至今）

转型一：HANA 和 S/4HANA（2010-2020）

2010 年，SAP 发布了一个叫 HANA 的东西，这是一个在内存中运行的数据库。

传统的数据库把数据存在硬盘上，读写很慢。HANA 把数据存在内存中，读写速度有数量级提升。这听起来像技术升级，但它的商业影响很深远。HANA 让 SAP 有了自己的数据库层，不再依赖 Oracle 和 Microsoft 的数据库。更重要的是，HANA 为 SAP 的新一代 ERP 产品 S/4HANA 提供了技术基础，因为数据跑在内存里，很多过去要多个系统协作完成的任务，现在一个系统就能搞定。

但问题来了：S/4HANA 要求客户从旧的 ECC 系统迁移过来。迁移意味着花几百万欧元、几年的时间、忍受业务中断的风险。客户不愿意。至今仍有 20,000+ 客户没有完成迁移，不是他们不想，是定制化太深、预算太紧、风险太高。

转型二：云 + AI（2020-至今）

从 2020 年开始，SAP 强迫自己从"卖永久软件"转向"卖云订阅"。这是一个商业模式的变化，不是技术变化。

2021 年推出的 RISE with SAP，把软件、基础设施、支持打包成一个订阅产品。2024 年花 $1.5B 收购 WalkMe，给它的 AI 助手装上"帮用户导航"的能力。2026 年密集收购 Reltio、Dremio、Prior Labs，补上数据层和 AI 层的能力。同年 5 月推出 Autonomous Enterprise（自主运营企业）愿景，说未来的企业不是人操作软件，而是人告诉 AI 想做什么，AI 来执行。

这个转型的核心意图可以概括为一句话：用 AI 重新包装云迁移。

客户不愿意因为"上云更便宜"这件事搬家，因为算下来真的不便宜。但如果不上云就没有 AI agent，不上云就享受不了"自主运营"呢？这是一个更有吸引力的理由。从客户反馈来看，SAP 的策略有在生效，43% 的客户把 SAP 的 AI 公告作为他们 ERP 策略调整的主要因素。但信任问题也在积累，72% 的客户对许可证条款持保留态度。

三个故事之后：它的基因是什么

三代产品、三阶段的历史，背后有四条反复出现的线索，这就是 SAP 的基因。

第一，它擅长做"地基"，不擅长做"门面"。 SAP 的产品从来不是界面最好看的。SAP 擅长的是让几十个国家、几千个用户、几万张表的财务数据在同一个系统里"跑对"。它不擅长让你打开系统时心情愉悦，但它擅长让你月底结账时数字对得上。

第二，它擅长"锁定"，不擅长"放手"。 SAP 的每一次产品迭代，都有意或无意地让客户更离不开它。从 R/2 到 R/3 到 S/4HANA 到 BTP，每一步都在加一层的锁定深度。客户发现走不了，这个事实本身就是 SAP 最大的护城河。

第三，它擅长"渐进改良"，不擅长"颠覆创新"。 SAP 从不推倒重来。它总是在已有的地基上加一层新的东西，R/3 在 R/2 的流程模型上加了一层新技术架构，HANA 在传统数据库上加了一层内存计算，现在 Autonomous Enterprise 在 S/4HANA 上加了一层 AI 层。这样做的好处是客户不用一次性革命，坏处是永远甩不掉历史包袱。

第四，它擅长"对公不对私"。 SAP 的客户是企业，不是个人。这意味着它的产品不需要讨个体用户的喜欢，只需要讨好 CFO 和 CIO。这是一个非常重要的基因，SAP 不需要像苹果那样让你"爱上"它的产品，它只需要让你"不得不"用它。

第三章：它的位置，和谁竞争、谁在威胁它

SAP 是全球最大的企业管理软件公司，不是最大的软件公司（微软更大）。在 ERP 市场中，它的份额约 24%，是第一名。第二名 Oracle 约 17%，第三名微软 Dynamics 约 6%。

但份额和竞争格局不是一回事。SAP 的竞争不是"谁把谁取代"，在 ERP 领域，大企业换掉 SAP Core 几乎不可能发生。竞争发生在更微观的层面：

和 Oracle 的竞争是最直接的。两家在财务和供应链模块上正面碰撞。Oracle 的差异化在于它有完整的云基础设施（OCI），而 SAP 的云跑在 AWS/Azure/GCP 上，这意味着 Oracle 可以告诉客户"我们的所有东西都是自己做的"，SAP 不能说这句话。但 SAP 在制造业的深度是 Oracle 追不上的。

和微软的竞争正在加剧。微软 Dynamics 365 + Copilot + Azure 的"三件套"对中型企业非常有吸引力。SAP 在 SME 市场面临最大的压力来自微软。

和 Workday 的竞争是模块级的。Workday 在 HR 领域比 SAP 的 SuccessFactors 做得更好。很多大企业的配置是"用 SAP 管财务和供应链，用 Workday 管 HR"，这已经成了一个相当主流的格局。

和 ServiceNow 的竞争是新兴的。ServiceNow 从 IT 服务管理切入业务流程自动化。它和 SAP 不在同一层竞争，但 ServiceNow 正在向上走，SAP 的 AI agent 向下覆盖，未来两年它们会在流程自动化层越来越近。

和 Salesforce 的竞争，SAP 在这个领域长期打不过。它的 CX（客户体验）产品渗透率不到 10-15%，几乎放弃了正面竞争。

对于客户来说，选择从来不是单一的"SAP 还是 Oracle"。更常见的情况是：你用 SAP 做财务和供应链，用 Workday 做人事，用 ServiceNow 做 IT 管理，用 Salesforce 做销售。 SAP 是一个核心组件，但不是全部。其他的模块产品各有自己的优势领域。

第四章：它要走向哪里

SAP 公开宣布的未来方向是 Autonomous Enterprise（自主运营企业），企业核心流程由 AI agent 自主执行，人定方向，AI 来完成。

为了实现这个方向，SAP 正在做三件事：

第一，用 AI 重新包装云迁移。 旧的"上云更便宜"叙事已经失效，客户算完账发现长期总成本没降。SAP 现在用的新理由是"不上云就没有 AI agent"。这是它在 Sapphire 2026 上宣布的条件：ECC 和 on-prem 客户必须承诺将至少 50% 的维护费转向云，才能获得部分 AI 能力。这是在用 AI 作为最后的推力，把不想迁的客户推上云。

第二，用收购补自己来不及做的事。 SAP 在 2024-2026 年密集收购了 WalkMe、Reltio、Dremio、Prior Labs，分别解决 AI 的交互能力、数据统一、非 SAP 数据接入、表格 AI 模型的问题。这些收购的共同特征是：SAP 判断这些方向至关重要，但自己从零做太慢了。

第三，把 AI agent 变成新的收入线。 目前 RISE 客户送 3 个免费 Joule agent，超额按需收费。如果这个模式跑通，SAP 可以在不增加客户数的情况下提升每客户 ARR。未来可能走向按任务量收费，用多少 AI agent 能力，付多少钱。

这三个动作指向的未来是：SAP 从一家卖 ERP 应用的公司，变成一个卖平台能力的公司。它的增长引擎会从"新客户签约"变成"存量客户多买模块 + AI agent 超额使用费"。

这个方向是否成立，取决于一个问题：客户还信不信 SAP。 故事很好听，但过去几年的迁移冲突、抵扣条款的争议已经让很多客户带着戒备在看 SAP 的每一个新动作。

理解 SAP 的线头

如果你想理解 SAP 这家公司，不需要记住它所有的产品、收购和财务数据。你只需要记住一个视角：

核心视角

SAP 最特殊的不是它的产品多好，而是它"加层"的发展方式。自 1972 年以来的每一次重大变化，它都在现有的地基上加一层新的东西，从不推倒重来。这个方式让它在三次关键转型中都没有丢失客户，但也让它背上了越来越重的历史包袱，核心系统变得极其复杂、客户越来越难升级、每次转型都需要更长时间才能完成。

这个视角解释了很多看起来矛盾的现象：

  • 为什么 SAP 的产品界面至今还像 1990 年代的东西？因为加层太多，每一层都有兼容旧系统的负担
  • 为什么 SAP 每次转型都显得很慢？因为它的转型方式不是换赛道，而是把新轨道铺在旧轨道上面
  • 为什么客户对 SAP 的感情是"又恨又离不开"？因为加层的方式让客户不用一次性变化太大，但也让客户越来越深地被锁定在系统里
  • 为什么 SAP 的护城河极深但增长空间有限？因为加层的优势是稳固，劣势是速度

这个加层逻辑是 SAP 区别于 Oracle、Salesforce、Microsoft 的独特之处。 它的竞争对手可以选择换跑道、做颠覆式创新，但 SAP 做不到。它的客户和历史决定了它只能一层一层往上加。这意味着 SAP 的任何新方向，无论是自主运营企业还是 AI agent，都会比市场期待的速度慢、但比市场想象的稳固。
