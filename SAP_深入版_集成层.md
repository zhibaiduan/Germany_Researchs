# 第二部分：集成层

---

## 六、SAP BTP 深入

> BTP 是理解现代 SAP 集成架构的核心。所有"SAP 和外部系统怎么连"的问题，答案几乎都经过 BTP。

### 6.1 BTP 是什么

SAP BTP（Business Technology Platform）是 SAP 的云平台，定位是 SAP 生态的"神经系统"——负责连接核心 ERP、外围云产品、外部系统，以及承载所有扩展开发。

BTP 不是一个单一产品，而是一组服务的集合，按功能分为四个主要领域：

```
┌─────────────────────────────────────────────────────┐
│                   SAP BTP                           │
├──────────────┬──────────────┬──────────────┬────────┤
│  数据与分析  │  应用开发    │  集成        │  AI    │
│  Data & Ana  │  App Dev     │  Integration │  & ML  │
├──────────────┼──────────────┼──────────────┼────────┤
│ SAP HANA     │ SAP Build    │ Integration  │ SAP AI │
│ Cloud        │ (低代码)     │ Suite        │ Core   │
│              │              │              │        │
│ SAP Data     │ SAP Build    │ Event Mesh   │ Joule  │
│ Sphere       │ Code         │              │        │
│              │ (Pro代码)    │ API Mgmt     │        │
│ Analytics    │              │              │        │
│ Cloud        │ SAP Build    │              │        │
│              │ Work Zone    │              │        │
└──────────────┴──────────────┴──────────────┴────────┘
```

### 6.2 四个核心服务详解

**Integration Suite（集成套件）**

这是 BTP 里和"SAP 与外部系统集成"最直接相关的服务，包含以下组件：

*Cloud Integration（原 HCI/CPI）*
SAP 自己的集成中间件，类似 MuleSoft 或 Azure Integration Services 的功能。用于在 SAP 系统和外部系统之间传输和转换数据。支持多种协议（REST、SOAP、SFTP、JDBC 等），内置大量预构建的集成包（Integration Packages），覆盖 Salesforce、Workday、Microsoft 365 等主流系统。

典型使用场景：Salesforce 成交订单自动同步到 SAP SD 创建销售订单；SAP HCM 员工数据同步到 SuccessFactors；外部供应商发票通过 EDI 传入 SAP MM。

*API Management*
管理 SAP 暴露给外部的 API。功能包括：API 的访问控制（谁能调用）、流量限制（防止过载）、监控（调用日志、错误率）、API 版本管理。

对 ISV 的意义：你的产品如果需要调用 SAP 的数据，通常会通过 API Management 层。API Management 是 CIO 控制"哪些外部系统能访问 SAP 数据"的门控。

*Event Mesh*
SAP 的事件总线服务。支持事件驱动架构：当 SAP 里发生某个业务事件（比如销售订单创建），Event Mesh 自动向所有订阅了这个事件的系统发送通知，不需要轮询。

这是现代集成架构的重要方向，但在德国企业中采用率还不高，很多企业仍在用传统的批处理或实时 API 调用模式。

*Open Connectors*
预构建的第三方系统连接器，覆盖 170+ 个常见 SaaS 应用。理论上可以快速连接 Salesforce、HubSpot、ServiceNow 等系统，不需要从头开发集成。实际上预构建连接器的成熟度参差不齐，复杂场景仍然需要定制。

**SAP Build（应用开发）**

面向业务用户和专业开发者的低代码/高代码开发平台，包含三个工具：

*SAP Build Apps（原 AppGyver）*
低代码移动和 Web 应用构建工具。业务用户可以拖拉拽构建简单应用，通过 BTP API 连接 SAP 数据，不需要写代码。典型场景：为仓库工人构建一个移动收货确认 App，替代 SAP GUI 界面。

*SAP Build Code*
面向专业开发者的代码开发环境，集成了 Joule AI 辅助编码。基于 VS Code，支持 CAP（Cloud Application Programming Model），是 SAP 推荐的扩展开发方式（Clean Core 理念下的标准路径）。

*SAP Build Work Zone*
企业门户和工作中心，把 SAP Fiori 应用、第三方应用、文档、通知整合在一个统一界面里。类似企业内网门户，但与 SAP 数据深度集成。

**SAP HANA Cloud 和 Data Sphere（数据层）**

*SAP HANA Cloud*
云端 HANA 数据库服务。如果客户的 S/4HANA 跑在云上，数据库就是 HANA Cloud。也可以作为独立的分析数据库使用。

*SAP Datasphere（原 Data Warehouse Cloud）*
SAP 的数据仓库和数据编织（Data Fabric）平台。把 SAP 数据和非 SAP 数据整合在一起，为分析和 AI 提供统一数据层。对于需要跨 SAP 和 Salesforce、Workday 等系统做综合分析的场景，Datasphere 是 SAP 推荐的解决方案。

**SAP AI Core（AI 层）**

BTP 上的 AI 开发和部署平台。企业可以在 AI Core 上训练和部署自定义 AI 模型，调用 SAP 的预构建 AI 服务（包括 Joule 背后的模型），以及集成第三方 AI 服务。

### 6.3 BTP 的部署选项和影响

BTP 本身是云服务，运行在超大规模云厂商（AWS、Azure、Google Cloud）上，SAP 在全球多个区域有数据中心，包括法兰克福（德国/欧盟数据主权合规）。

对德国客户的意义：如果客户要求数据留在欧盟，BTP 的法兰克福区域可以满足这个要求。但如果客户要求数据在自己的数据中心（On-Premise 部署），BTP 的云服务就无法使用，需要用 SAP Process Orchestration（PO，BTP 的本地部署前身）作为替代。

**这是一个重要的判断点**：如果你的集成方案依赖 BTP，需要先确认客户是否允许使用云服务，以及是否有数据出境的限制。

### 6.4 BTP 的实际采购和成本

BTP 按服务消费量计费，有以下几种方式：

消费制（CPEA：Cloud Platform Enterprise Agreement）：客户预购一定量的"积分"（Credits），按实际使用的服务消耗积分。适合使用量不稳定的场景。

订阅制：按月/年固定费用使用特定服务，适合使用量可预测的场景，通常比消费制便宜。

捆绑许可：购买 S/4HANA 或其他 SAP 云产品时，通常会捆绑一定量的 BTP 积分。很多客户有 BTP 积分但不知道怎么用，这是一个切入点。

**对你的意义**：很多已经购买了 S/4HANA Cloud 或 SuccessFactors 的客户，已经有了未使用的 BTP 积分。如果你的产品可以在 BTP 上运行或通过 BTP 集成，"用你们已经付过钱的 BTP 积分来部署"是一个有说服力的采购理由。

## 七、SAP API 层：OData、BAPI、IDoc

> 这三种技术是 SAP 对外暴露功能的主要方式。不需要自己写代码，但需要能在和客户 IT 架构师的对话中准确使用这些概念，判断集成方案的技术路径。

### 7.1 三种 API 技术的定位

```
现代程度
新 │  OData API（RESTful，S/4HANA 主推）
   │  BAPI（面向对象，ECC/S/4均支持）
旧 │  IDoc（文档交换，EDI场景）
   └──────────────────────────────→ 适用场景广度
```

### 7.2 OData API

**是什么**
OData（Open Data Protocol）是一种基于 REST 的标准协议，SAP 在 S/4HANA 和 Fiori 架构中大量使用。所有 Fiori 应用背后都是 OData API，这意味着任何 Fiori 能做的事，通过 OData API 都能程序化实现。

**技术特点**
- 基于 HTTP，使用 GET/POST/PUT/DELETE 标准方法
- 返回 JSON 或 XML 格式，现代开发者熟悉
- 支持过滤、排序、分页等查询参数
- SAP 在 API Business Hub（api.sap.com）上发布了所有标准 OData API 的文档

**适用场景**
- 实时数据查询（查询库存状态、销售订单信息）
- 创建或更新 SAP 业务对象（创建销售订单、更新客户主数据）
- 与现代 SaaS 系统集成（这些系统的开发者熟悉 REST API）

**限制**
- 只在 S/4HANA 和较新版本的 ECC 中完整支持，老版 ECC 的 OData 覆盖范围有限
- 标准 OData API 不一定覆盖所有业务场景，复杂场景可能需要客户自定义开发 OData 服务
- 性能：大批量数据传输用 OData 不是最优选择

**你需要知道的一句话**：OData 是现代 SAP 集成的首选方式，如果客户在 S/4HANA 上，优先考虑 OData。

### 7.3 BAPI

**是什么**
BAPI（Business Application Programming Interface）是 SAP 从 1990 年代就有的标准函数接口，用 ABAP 编写，通过 RFC（Remote Function Call）协议调用。

**技术特点**
- 通过 RFC 协议调用，不是 HTTP
- 需要 SAP 专用的连接库（SAP JCo for Java，SAP NCo for .NET，pyrfc for Python）
- 有严格的输入/输出参数定义，稳定可靠
- SAP 提供了几千个标准 BAPI，覆盖大部分核心业务操作

**适用场景**
- ECC 环境中的集成（OData 覆盖不足时）
- 需要调用 SAP 标准业务逻辑的场景（BAPI 内置了完整的业务规则校验）
- 批量数据处理（BAPI 支持批量调用，性能比 OData 好）

**限制**
- 需要安装 SAP 专用连接库，增加了技术复杂度
- RFC 协议不是标准 HTTP，防火墙配置和网络安全要求更复杂
- 在 Public Cloud 部署的 S/4HANA 中，直接 RFC 调用受到限制，需要通过 BTP 的集成服务转换

**常见 BAPI 举例**：
- `BAPI_SALESORDER_CREATEFROMDAT2`：创建销售订单
- `BAPI_PO_CREATE1`：创建采购订单
- `BAPI_MATERIAL_SAVEDATA`：创建或更新物料主数据
- `BAPI_ACC_DOCUMENT_POST`：过账财务凭证

**你需要知道的一句话**：如果客户在 ECC 上，BAPI 通常是集成的主要技术路径。如果听到客户说"我们用 RFC 调用"，他们说的就是 BAPI。

### 7.4 IDoc

**是什么**
IDoc（Intermediate Document）是 SAP 的文档交换格式，主要用于 EDI（电子数据交换）场景——企业和供应商、客户、银行之间的标准化业务文档传输。

**技术特点**
- 基于文件的异步传输，不是实时调用
- 有严格的结构定义，每种业务文档类型（订单、发票、发货通知）有对应的 IDoc 类型
- 传输可靠性高，有完整的状态追踪和错误处理机制
- 可以通过 SFTP、AS2、HTTP 等多种方式传输 IDoc 文件

**适用场景**
- 与供应商系统的 EDI 集成（发送采购订单、接收发货通知）
- 与银行的电子对账单交换（MT940 银行对账单导入）
- 大批量业务文档的异步传输
- 跨 SAP 系统之间的数据同步（ALE，Application Link Enabling）

**限制**
- IDoc 格式复杂，解析和生成需要专业知识
- 异步传输意味着不适合需要实时响应的场景
- 在现代云集成场景中，IDoc 逐渐被 OData 和事件驱动架构替代，但在制造业和供应链场景中仍然大量使用

**常见 IDoc 类型举例**：
- `ORDERS05`：采购订单
- `INVOIC02`：发票
- `DESADV`：发货通知（ASN，Advance Shipping Notice）
- `REMADV`：付款通知

**你需要知道的一句话**：如果客户提到 EDI 集成或和供应商系统对接，IDoc 几乎肯定会出现。

### 7.5 三种技术的选择决策树

```
需要集成的客户环境是？
    │
    ├─ S/4HANA（On-Prem 或 Cloud）
    │       │
    │       ├─ 实时查询或操作 → OData API（首选）
    │       ├─ 批量处理 → OData Batch 或 BAPI
    │       └─ EDI/供应链文档交换 → IDoc
    │
    └─ ECC（老版本）
            │
            ├─ 实时操作 → BAPI（OData 覆盖有限）
            ├─ 批量处理 → BAPI 或直接数据库（不推荐）
            └─ EDI/供应链文档交换 → IDoc
```

### 7.6 一个你必须知道的额外概念：CDS View

CDS（Core Data Services）View 是 S/4HANA 中定义数据视图的方式，也是 OData API 的底层基础。理解 CDS View 的原因：

当客户说"我们需要一个自定义数据接口"，在 S/4HANA 里的标准做法是创建一个 CDS View 暴露所需数据，然后在此基础上生成 OData API。这比直接写 ABAP 代码更符合 Clean Core 理念，也更容易维护。

听到 CDS View 时，你可以理解为"S/4HANA 里定义数据接口的标准方式"，不需要深入了解技术细节，但这个词在和 SAP 技术团队对话时会频繁出现。

## 八、集成模式与方案设计

> 技术选好了，还需要选对集成模式。不同的业务场景需要不同的集成模式，选错了即使技术实现了也会带来运营问题。

### 8.1 三种基本集成模式

**模式一：实时同步（Real-time / Synchronous）**

```
系统A ──── 请求 ────→ 系统B
系统A ←─── 响应 ──── 系统B
（等待响应，毫秒到秒级）
```

特点：系统A发出请求后等待系统B响应，然后继续处理。数据一致性强，但系统B的性能直接影响系统A。

适用场景：
- 用户在界面上点击"检查库存"，需要立即知道结果
- 创建销售订单时实时校验客户信用额度
- 支付处理时实时确认

技术实现：OData API 调用、BAPI 同步调用

风险：系统B出现问题（延迟、宕机），系统A也会受影响。需要设计超时处理和降级策略。

**模式二：异步消息（Asynchronous / Message-based）**

```
系统A ──── 消息 ────→ 消息队列 ────→ 系统B
系统A 继续处理，不等待系统B响应
（解耦，秒到分钟级）
```

特点：系统A把消息放入队列后立即返回，系统B异步处理消息。系统之间解耦，一方出问题不影响另一方。

适用场景：
- Salesforce 成交订单，异步触发 SAP 创建销售订单（销售人员不需要等 SAP 处理完才能继续工作）
- SAP 发货完成后，异步通知 CRM 更新订单状态
- 高峰期的大量事件处理（流量削峰）

技术实现：BTP Event Mesh、SAP Integration Suite 的消息队列功能

风险：消息处理失败时需要有重试机制和错误监控；消息顺序不能保证（需要业务逻辑处理乱序情况）。

**模式三：批处理（Batch Processing）**

```
系统A ──── 数据文件 ────→ 传输（SFTP/API）────→ 系统B
（定时执行，分钟到小时级）
```

特点：定时（每小时、每天）把积累的数据打包传输。效率高，对网络要求低，但数据有延迟。

适用场景：
- 每天晚上把 ERP 里的财务数据导出到 BI 系统
- 每小时把 CRM 的客户更新同步到 SAP
- 银行对账单每天导入 SAP FI

技术实现：IDoc 文件传输、BTP Integration Suite 的批处理流、SFTP 文件交换

风险：数据延迟；如果批处理失败，需要完整的错误处理和重跑机制；数据量大时需要考虑性能。

### 8.2 选择集成模式的决策框架

| 考量因素 | 实时同步 | 异步消息 | 批处理 |
|---------|---------|---------|--------|
| 数据时效性要求 | 秒级 | 秒到分钟级 | 分钟到小时级 |
| 数据量 | 小（单条记录） | 中（事件流） | 大（批量记录） |
| 系统解耦需求 | 低 | 高 | 高 |
| 错误处理复杂度 | 低 | 中 | 中 |
| 对系统B可用性依赖 | 高 | 低 | 低 |
| 实现复杂度 | 低 | 中 | 中 |

**一个判断原则**：能用异步的尽量不用实时同步。实时同步的系统耦合度高，一个系统出问题会级联影响另一个。在企业集成场景中，系统可靠性不如互联网产品，异步解耦是更稳健的设计。

### 8.3 五个典型集成场景的标准方案

**场景一：Salesforce → SAP（CRM 订单到 ERP）**

业务需求：销售在 Salesforce 成交，需要在 SAP SD 模块创建销售订单。

推荐方案：异步消息模式

```
Salesforce（机会关闭/订单确认）
    ↓ Salesforce Platform Event（事件触发）
BTP Integration Suite（数据转换和路由）
    ↓ 数据映射（Salesforce 字段 → SAP SD 字段）
SAP S/4HANA（通过 OData API 创建销售订单）
    ↓ 处理结果回传
BTP Integration Suite（错误处理和状态更新）
    ↓
Salesforce（更新订单状态）
```

关键挑战：
- 字段映射复杂（Salesforce 的客户 ID 需要映射到 SAP 的客户主数据）
- 错误处理（如果 SAP 创建失败，如何通知销售团队）
- 数据重复（同一个订单不能在 SAP 里创建两次）

**场景二：SAP HR → 外部 HR 系统（员工数据同步）**

业务需求：SAP HCM 是员工主数据的 System of Record，需要定期同步到 Workday 或 Personio。

推荐方案：批处理（每天一次或每小时一次）

```
SAP HCM（员工主数据变更检测）
    ↓ IDoc 或 OData 导出变更记录
BTP Integration Suite（数据格式转换）
    ↓ 字段映射（SAP 组织结构 → 目标系统结构）
Workday / Personio（API 更新员工数据）
    ↓ 处理结果
BTP Integration Suite（错误日志和告警）
```

关键挑战：
- 初始数据加载（首次同步所有历史数据）
- 增量同步（只同步变更的记录，不全量传输）
- 数据冲突处理（两个系统都有人修改同一条记录时以谁为准）

**场景三：外部系统 → SAP FI（财务凭证写入）**

业务需求：电商平台的销售数据每天汇总后需要写入 SAP FI 生成财务凭证。

推荐方案：批处理

```
电商平台（每天生成销售汇总文件）
    ↓ SFTP 或 API 传输
BTP Integration Suite（数据验证和转换）
    ↓ 生成 SAP 标准格式（BAPI 或 IDoc）
SAP S/4HANA FI（BAPI_ACC_DOCUMENT_POST 过账凭证）
    ↓ 过账结果
BTP Integration Suite（成功/失败记录，错误通知）
```

关键挑战：
- 财务数据的准确性要求极高，任何错误都会影响账务
- 科目映射（电商平台的收入分类需要映射到 SAP 的会计科目）
- 已过账凭证无法直接删除，错误处理需要反向凭证冲销机制

**场景四：SAP → BI 工具（数据分析同步）**

业务需求：把 SAP 的财务和运营数据定期同步到 Power BI 或 Tableau 做分析。

推荐方案：批处理（每天一次）或实时（通过 OData 直连）

```
方案A：批处理（推荐，稳定性高）
SAP S/4HANA（CDS View 定义数据视图）
    ↓ 定时导出（每天凌晨）
SAP Datasphere 或 Azure Data Factory（数据存储和转换）
    ↓
Power BI / Tableau（连接数据源，构建报表）

方案B：实时直连（简单但有性能风险）
Power BI → OData 直连 SAP S/4HANA
（适合数据量小、查询频率低的场景，
 大量并发查询可能影响 SAP 性能）
```

关键挑战：
- SAP 不应该承受大量 BI 查询的压力（影响业务系统性能），方案A的数据中间层更稳健
- 数据时效性 vs 系统压力的权衡
- 历史数据的处理（几年的历史财务数据量可能很大）

**场景五：供应商 EDI 集成**

业务需求：与供应商实现电子数据交换——发送采购订单，接收发货通知和发票。

推荐方案：IDoc + EDI 中间件

```
SAP MM（生成采购订单）
    ↓ IDoc（ORDERS05 类型）
EDI 中间件（SAP Integration Suite 或 专业 EDI 平台如 OpenText）
    ↓ 格式转换（IDoc → EDIFACT / X12 标准格式）
    ↓ 传输（AS2 / SFTP）
供应商系统
    ↓ 回传发货通知（DESADV）
EDI 中间件（格式转换）
    ↓ IDoc（DESADV 类型）
SAP MM（更新收货状态）
```

关键挑战：
- 每个供应商的 EDI 标准和字段映射都不同，需要逐一配置
- 供应商的技术能力参差不齐，有些供应商只能接受邮件 PDF 而不是 EDI
- 错误处理：发货通知数量和采购订单数量不匹配时的处理逻辑

## 九、集成项目的常见坑

> 技术方案设计好了，项目执行中仍然有大量风险点。这一章是经验的浓缩，帮你提前识别问题，在客户面前建立专业可信度。

### 9.1 技术层面的坑

**坑一：忽视 SAP 版本差异**

同样叫"SAP ECC"，不同客户的 ECC 版本可能相差十几年（ECC 5.0 vs ECC 6.0 EHP8），支持的 API 和功能差异很大。项目开始前必须确认：

- SAP 的精确版本号（比如 S/4HANA 2022 vs S/4HANA 2023）
- 已安装的 Enhancement Package（EHP）
- 部署形态（On-Prem / Private Cloud / Public Cloud）

不同版本可用的 OData API 不同，自定义开发的限制不同，BTP 的可用服务也可能不同。用错误的版本假设做出来的集成方案可能在客户环境里根本跑不起来。

**坑二：低估主数据质量问题**

集成项目最常见的延期原因之一。主数据（客户主数据、物料主数据、供应商主数据）是集成的基础——如果 Salesforce 里的客户记录和 SAP 里的客户记录对不上，整个 CRM-ERP 集成就无法自动运行。

典型问题：
- 同一个客户在 Salesforce 和 SAP 里有不同的 ID 和名称格式
- SAP 里有大量重复的物料主数据（同一个物料有多个物料号）
- 供应商银行账户信息不完整或格式不标准

**在项目开始时必须做的事**：进行主数据质量评估（Data Quality Assessment），量化问题的规模，把主数据清洗工作纳入项目计划。不做这步，后面一定会付出更大代价。

**坑三：网络和防火墙问题被低估**

On-Prem SAP 系统通常部署在企业内网，有严格的防火墙规则。从外部系统（无论是 BTP 还是你的 SaaS 产品）连接到 On-Prem SAP，需要：

- 开放特定端口（RFC 连接用 33xx 端口，HTTP 用 443）
- 配置 SAP Cloud Connector（BTP 和 On-Prem SAP 之间的安全通道）
- 通过企业 IT 安全团队的审批

这个审批流程在大型企业里可能需要 4-8 周。很多集成项目在技术开发完成后，才发现网络配置没有做，导致无法部署，时间表被迫推迟。**在项目启动时就应该同步启动网络配置的审批流程。**

**坑四：SAP Cloud Connector 的配置复杂度**

SAP Cloud Connector（SCC）是 BTP 连接 On-Prem SAP 的必要组件，安装在客户内网，建立从内网到 BTP 的安全隧道（反向连接，不需要开放入站端口）。

常见问题：
- SCC 安装在哪台服务器上？这台服务器的网络访问权限如何？
- SCC 的高可用性配置（主备模式），避免 SCC 成为单点故障
- SCC 版本更新策略（SAP 定期发布更新，需要有人维护）
- 多套 SAP 系统（开发/测试/生产）各需要配置独立的 SCC 连接

**坑五：错误处理和监控设计不足**

很多集成项目把 80% 的精力放在"成功路径"（Happy Path）上，对错误处理设计不足。在生产环境中，错误是常态而不是例外：

- 网络超时
- SAP 系统维护窗口期间不可用
- 数据校验失败（比如 SAP 里找不到对应的客户记录）
- 消息重复（同一条数据被处理了两次）

每种错误都需要有明确的处理逻辑：是自动重试，还是进入人工处理队列，还是发送告警？没有设计好错误处理，生产环境出问题时会陷入"数据黑洞"——不知道哪些数据处理成功了，哪些失败了。

**必须在设计阶段就确定的错误处理策略**：
- 重试次数和间隔（比如：失败后每隔 5 分钟重试，最多重试 3 次）
- 告警机制（谁收到错误通知，通过什么渠道）
- 死信队列（多次重试后仍然失败的消息如何处理）
- 幂等性设计（同一条消息被处理两次，不会产生重复数据）

### 9.2 数据层面的坑

**坑六：字段映射的隐藏复杂度**

表面上看，字段映射（把系统A的字段对应到系统B的字段）是简单的工作。实际上，几乎每个字段背后都有业务逻辑：

- 日期格式：Salesforce 用 ISO 8601，SAP 内部用 YYYYMMDD，显示格式还可能是 DD.MM.YYYY（德国格式）
- 货币：Salesforce 存美元，SAP 可能需要欧元，还需要处理历史汇率
- 数量单位：Salesforce 里的"1000"可能是件数，SAP 里可能需要转换为千克
- 枚举值映射：Salesforce 的订单状态"Closed Won"需要映射到 SAP SD 的哪个状态？
- 组织结构：Salesforce 里的销售区域需要映射到 SAP SD 的销售组织/分销渠道/产品组三级结构

**一个实用建议**：在项目开始时建立一份完整的字段映射文档（Field Mapping Specification），每个字段的映射规则、转换逻辑、边界情况都要明确定义，并由业务方确认。这份文档后来会成为开发和测试的唯一依据。

**坑七：历史数据迁移 vs 持续集成混淆**

集成项目通常包含两个不同的工作流：
1. 历史数据迁移（一次性）：把过去的历史数据从旧系统迁移到新系统
2. 持续集成（长期运行）：新产生的数据自动在两个系统之间同步

这两个工作流的技术需求、数据量、错误处理逻辑完全不同，但经常被混为一谈。必须在项目计划中明确区分：历史数据迁移什么时候做、由谁负责、用什么工具；持续集成的架构是什么、在哪里运行、谁来维护。

### 9.3 组织层面的坑

**坑八：SAP 系统的变更审批流程**

在生产环境的 SAP 系统里做任何变更（开放一个 API、调整一个权限、安装一个插件），都需要经过变更管理流程（Change Management Process）：提交变更申请 → 评估影响 → 审批 → 测试 → 部署。

在大型企业里，这个流程可能需要 2-6 周。如果你的集成方案需要在 SAP 里开放 10 个 API，每个 API 的开放都是一个独立的变更申请，整个过程可能需要几个月。

**在项目启动时就要做的事**：梳理所有需要在 SAP 侧做的变更（开放哪些 API、创建哪些技术用户、调整哪些权限），一次性提交所有变更申请，不要等到开发完成才开始走审批。

**坑九：系统所有权和维护责任不清晰**

集成系统上线后，谁负责维护？当集成出问题时，谁来排查？这个问题在项目阶段经常没有明确答案，导致上线后出现问题时各方互相推诿。

常见的责任模糊区域：
- BTP Integration Suite 的流程出错了，是 SAP 团队负责还是集成团队负责？
- Salesforce 侧的数据格式变化导致集成失败，是 Salesforce 团队的问题还是集成团队的问题？
- 某个第三方 API 不可用导致集成中断，SLA 责任在哪里？

**在项目交付物中必须包含的内容**：RACI 矩阵（谁负责、谁审批、谁咨询、谁知情），明确每类问题的第一响应人和升级路径。

**坑十：测试环境和生产环境的差异**

SAP 项目通常有三套环境：开发（DEV）、质量保证/测试（QAS）、生产（PRD）。集成测试在 QAS 环境通过，不代表在 PRD 环境也能通过：

- PRD 的数据量比 QAS 大几个数量级，性能问题只在 PRD 才能暴露
- PRD 的网络配置、防火墙规则、权限设置可能和 QAS 不同
- PRD 的主数据质量可能比 QAS 更差（QAS 通常用的是清洗过的测试数据）

**必须在 QAS 测试阶段就要做的事**：用接近生产数据量的数据集做性能测试；确认 PRD 环境的网络和权限配置与 QAS 的差异；在 PRD 环境做切换前做一次完整的集成冒烟测试。

### 9.4 一张清单：集成项目启动时必须确认的十件事

```
□ 1. SAP 精确版本号和部署形态
□ 2. 主数据质量评估已完成，清洗工作已纳入计划
□ 3. 网络/防火墙变更申请已同步启动
□ 4. SAP Cloud Connector 安装服务器已确定
□ 5. 所有 SAP 侧变更申请已一次性提交
□ 6. 字段映射文档已完成并经业务方确认
□ 7. 历史数据迁移 vs 持续集成已明确区分
□ 8. 错误处理和监控策略已设计
□ 9. 测试/生产环境差异已梳理
□ 10. 维护责任 RACI 矩阵已确定
```
