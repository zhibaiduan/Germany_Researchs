# SAP 知识框架 v1.2

> 适用场景：在德国从事 B2B SaaS 产品或解决方案工作，需要理解 SAP 生态、与客户的 SAP 团队对话、判断产品切入点。
>
> 阅读建议：第三章"产品版图"是全文骨架，第四章"部署形态"决定了你能卖进去什么，建议先读这两章再读其他章节。

## 一、为什么存在

1972 年，五个从 IBM 离职的德国工程师在曼海姆创立 SAP。他们之前在 IBM 给企业做定制化财务软件，每个客户都要重新开发一套，效率极低。核心洞察是：企业的财务和业务流程大同小异，可以做成标准化软件，一套卖给所有人。

当时企业的现实：财务用一套系统记账，仓库用另一套管库存，销售用纸质单据，月底手动汇总对账，经常出错。每个部门的数据互相不认识。

SAP 的解法是把所有业务数据放进同一个系统，共享同一个数据库。财务记一笔收款，库存自动扣减，销售订单自动关闭，不需要人工传递。

**SAP 存在的根本原因：消灭企业内部的数据孤岛，让所有业务动作在同一个系统里留痕、联动、可查。**

## 二、发展关键节点

**1970s：押注实时处理**
当时主流是批处理，数据隔夜才能更新。SAP 第一代产品 R/1 做实时处理，数据录入即时可见。

**1980s：跟着制造业走**
德国是制造业大国，SAP R/2 专门针对汽车、化工、机械企业设计，拿下大众、西门子、拜耳。这批客户成了最有力的市场背书。

**1992：R/3 是真正转折点**
从大型机转向客户端-服务器架构，中型企业也能用得起 SAP。同年在纽约证交所上市，全球扩张开始。

**1990s 末：抓住 Y2K 窗口**
千年虫问题让全球企业更换老旧系统，SAP 借此拿下大量欧美客户。

**2000s：专注大企业市场**
Oracle、Microsoft 争夺中小企业，SAP 专注年收入 10 亿欧以上的大企业。这个客群黏性极高，一旦上线替换成本极大。

**2010s：收购弥补产品缺口**
2012 年收购 SuccessFactors（HR），2013 年收购 Hybris（电商），2014 年收购 Concur（差旅费用）。核心 ERP 覆盖不足的模块通过收购补齐，形成"云产品矩阵"。

**2015 至今：S/4HANA 战略转型**
把核心 ERP 迁移到内存数据库 HANA，处理速度大幅提升。推出云版本和现代界面 Fiori。宣布老版本 ECC 将于 2030 年停止官方支持，推动客户向新平台迁移。S/4HANA 迁移目前是德国企业最集中的 IT 预算方向，详见第十一章切入点框架。

## 三、SAP 今天的产品版图

> 这是理解后续所有内容的骨架。SAP 不是一个单一产品，是由四层构成的平台体系。

```
┌─────────────────────────────────────────────────┐
│                   界面层                         │
│   SAP GUI（经典桌面）/ SAP Fiori（现代浏览器）  │
├─────────────────────────────────────────────────┤
│               平台与集成层                       │
│   SAP BTP（Business Technology Platform）       │
│   API 管理 / 集成服务 / 扩展开发 / 低代码工具   │
├─────────────────────────────────────────────────┤
│               外围云产品层                       │
│   SuccessFactors（HR 人才管理）                  │
│   Ariba（外部采购协作网络）                      │
│   Concur（差旅与费用报销）                       │
│   SAP CX / C/4HANA（客户体验与 CRM）            │
├──────────────────────────────┬──────────────────┤
│   核心 ERP 层                │                  │
│   经典核心模块               │  演进中的模块    │
│   FI · CO · MM · SD · PP    │  HCM → SF 替代   │
│   （高度稳定，长期主导）     │  WM → EWM 升级   │
└──────────────────────────────┴──────────────────┘
```

**核心 ERP 层：经典核心模块 vs 演进中的模块**

FI、CO、MM、SD、PP 是 SAP 最稳定、最核心的模块，在 ECC 和 S/4HANA 中均处于主导地位，是财务、供应链、生产的 System of Record，几乎没有被替换的趋势。

HCM（人力资源）和 WM（仓储管理）则处于"演进中"的状态：
- HCM 的薪资计算和基础人事记录仍在核心 ERP 层，但招聘、绩效、学习等人才管理功能正在被 SuccessFactors 替代。很多企业目前是 HCM 和 SuccessFactors 并行，HCM 管"记录"，SuccessFactors 管"体验"。
- WM 在 S/4HANA 中已被 EWM（Extended Warehouse Management）替代，EWM 支持自动化仓储设备集成，能力更强。老版 WM 仍在 ECC 企业中广泛运行。

这个区分很重要：如果你的产品涉及 HR 或仓储场景，需要判断客户用的是老版模块还是新版云产品，集成方式和决策链完全不同。

**外围云产品层**
SAP 通过收购或自研形成的专业 SaaS 产品，各自有独立的系统和界面，通过 BTP 或 API 与核心 ERP 集成。它们不是 ERP 核心的一部分，而是专业能力的扩展。

**平台与集成层**
SAP BTP 是 SAP 的集成与扩展中枢，负责核心 ERP 与外围云产品之间的数据同步，也是 ISV 和客户自定义开发的主要入口。可以理解为"SAP 生态的神经系统"。

**界面层**
SAP GUI 是沿用多年的经典桌面界面，功能全面但操作复杂。SAP Fiori 是基于浏览器的现代界面，支持移动端，体验更友好，但并非所有功能都已迁移。许多企业两套界面并行使用。

**各层关系一句话**
核心 ERP 是数据底座，外围云产品是专业能力扩展，BTP 是连接桥梁，界面层是人与系统的交互入口。ISV 通常插在 BTP 层或界面层之上，填补 SAP 原生能力的空白。

## 四、SAP 部署形态

> 同样叫"SAP"，但部署形态不同，意味着完全不同的定制自由度、集成策略、合规要求和 ISV 接入方式。这是判断能否卖进去、怎么卖的关键变量。

### 四种主要部署形态

```
定制自由度
高 │  ECC On-Prem
   │  S/4 On-Prem
   │  S/4 Private Cloud
低 │  S/4 Public Cloud
   └──────────────────────────────→ 上线速度 / 维护便利性
```

**ECC On-Premise（本地部署老版 ERP）**

服务器在客户自己的数据中心，SAP 不直接管理。客户对系统有完全控制权，可以做任意深度的 ABAP 定制开发，可以修改标准代码，可以接入任何 ISV 工具。

这是德国大型企业最主流的现有形态。灵活但维护成本高，需要专职 IT 团队和 SAP Basis 管理员。所有 ECC 客户面临 2030 年迁移压力。

对你的意义：ISV 接入最自由，几乎没有技术限制；但变更审批流程复杂，CIO 对新工具的评估周期较长。

**S/4HANA On-Premise（本地部署新版 ERP）**

服务器仍在客户数据中心，但跑的是新版 S/4HANA。保留了 ECC On-Prem 的定制自由度，同时获得 HANA 数据库的性能提升和新功能。

这是从 ECC 迁移的主流路径之一，尤其是对数据主权要求高、定制化需求复杂的德国大型企业。迁移成本高，周期 2-4 年。

对你的意义：与 ECC On-Prem 类似，ISV 接入自由度高，但迁移项目本身是最大的切入窗口。

**S/4HANA Private Cloud（私有云）**

SAP 或合作伙伴管理的专用云环境，基础设施不与其他客户共享。客户不需要管理服务器，但仍可以做一定程度的定制开发（比 Public Cloud 自由，比 On-Prem 受限）。

迁移速度比 On-Prem 快，维护成本低。越来越多的中大型企业在 S/4HANA 迁移时选择这条路，尤其是有合规需求但又想降低 IT 运维负担的企业。

对你的意义：ISV 接入需要通过 BTP 或认证的 API，直接修改系统的空间受限，但 BTP 层的集成能力足够支撑大多数产品场景。

**S/4HANA Public Cloud（公有云）**

SAP 管理的多租户云环境，类似 SaaS。基础设施与其他客户共享，几乎不支持 ABAP 定制开发，功能以"干净核心"为主（Clean Core 策略：尽量不做定制，走标准流程）。

上线速度最快（几个月 vs 几年），维护成本最低，自动获得 SAP 最新功能更新。适合流程相对标准的中型企业，或者希望快速上线的新建公司。

德国大型企业几乎不用这个选项，因为定制化需求太复杂，合规要求也不允许数据放在多租户环境中。

对你的意义：ISV 接入受到严格限制，必须通过 SAP Store 认证和 BTP 集成，不能直接操作数据库层。但这个形态的客户更愿意接受新的 SaaS 工具，采购决策链也更短。

### 部署形态对照表

| 维度 | ECC On-Prem | S/4 On-Prem | S/4 Private Cloud | S/4 Public Cloud |
|------|------------|------------|------------------|-----------------|
| 服务器位置 | 客户数据中心 | 客户数据中心 | 专用云 | SAP 共享云 |
| 定制自由度 | 最高 | 高 | 中 | 低 |
| ABAP 开发 | 完全支持 | 完全支持 | 受限 | 几乎不支持 |
| ISV 接入方式 | 直接集成 | 直接/BTP | 主要通过 BTP | 必须通过 BTP/Store |
| 上线速度 | 慢（2-4年） | 慢（2-4年） | 中（1-2年） | 快（3-6个月） |
| 维护负担 | 高（自管） | 高（自管） | 低（SAP管） | 最低（SAP管） |
| 典型客户 | 德国大型企业 | 迁移中大企业 | 中大型企业 | 中型/新建企业 |
| 2030压力 | 高（必须迁移） | 无 | 无 | 无 |
| 数据主权 | 完全自控 | 完全自控 | 合同保障 | 共享环境风险 |

### 对德国市场的特殊影响

德国企业对数据主权的敏感程度在全球是最高的之一。GDPR、行业监管（金融、医疗）、以及企业文化上对数据控制权的重视，导致 Public Cloud 在德国大型企业中接受度很低。大部分德国 1000+ 人的企业在迁移时会选择 On-Prem 或 Private Cloud，而不是 Public Cloud。

这对你的产品策略有直接影响：如果你的产品只支持云端部署，进入德国大企业的阻力会比进入美国企业大得多，需要准备好私有化部署或欧盟数据中心的选项。

## 五、核心模块详解

> 本章描述核心 ERP 层内部的模块逻辑，适用于 ECC 和 S/4HANA。

### 主要模块说明

**FI（Financial Accounting）财务会计**
对外的法律财务记录。每一笔收入、支出、资产变动在此记账。资产负债表、利润表从这里生成。德国《商法典》强制要求完整会计记录，FI 是合规底线。数据一旦入账不可随意删改，只能通过规范的反向凭证更正。

**CO（Controlling）管理会计**
对内的成本管理分析。成本中心（哪个部门花了多少）、利润中心（哪条业务线赚了多少）、项目成本追踪。FI 记对外账，CO 做内部管理分析。在 S/4HANA 中 FI 和 CO 的数据结构已合并，实时同步，消除了 ECC 时代月底对账的步骤。

**MM（Materials Management）物料管理**
采购到收货的全流程。采购申请、询价、采购订单、收货确认、发票校验、付款触发。核心内控机制是三单匹配：采购订单 + 收货单 + 供应商发票金额必须一致，才能放行付款。

**SD（Sales & Distribution）销售分销**
从报价到收款。报价单、销售订单、发货、开票、应收账款管理。前端销售人员通常在 CRM（如 Salesforce）里工作，SD 是后端执行层，负责把商业承诺转化为系统记录和物流动作。

**PP（Production Planning）生产计划**
制造业专用模块。生产订单管理、物料需求计划（MRP）、产能规划。制造业客户对此模块高度依赖，是 SAP 在制造业难以被替代的核心原因之一。

**HCM（Human Capital Management）人力资源**
员工主数据、薪资计算、考勤、组织架构管理。注意这是"演进中的模块"：HCM 负责薪资执行和基础人事记录（System of Record），但招聘、绩效、学习等人才管理流程越来越多地由 SuccessFactors 负责。很多企业目前是两者并行，HCM 管记录，SuccessFactors 管体验。

**WM / EWM（Warehouse Management）仓储管理**
仓库内部精细化操作：货位管理、拣货路径优化、移库、盘点。MM 管"有没有货"，WM/EWM 管"货在哪个货架第几层"。在 S/4HANA 中，WM 已被能力更强的 EWM（Extended Warehouse Management）替代，支持自动化仓储设备集成。老版 WM 仍在大量 ECC 环境中运行。

### 模块间数据流转

**销售链（从客户下单到收款）**

```
客户下单
→ SD 创建销售订单
→ MM 检查库存（不足则触发 PP 生产计划）
→ PP 触发 MM 采购原材料
→ WM/EWM 原材料入库 / 成品入库
→ SD 确认发货，WM/EWM 执行出库
→ FI 自动生成应收账款
→ 收款后 FI 核销
→ CO 实时记录利润数据
```

**采购链（从采购需求到付款）**

```
MM 发起采购申请
→ 审批通过，生成采购订单
→ 供应商发货，WM/EWM 收货入库
→ MM 三单匹配（采购订单 + 收货单 + 发票）
→ 匹配通过，FI 生成应付账款
→ 付款后 FI 核销
→ CO 记录采购成本
```

**关键规则**
- SD 和 MM 是业务触发层，负责生成业务单据
- FI 是最终记账层，任何业务动作必须在此留下财务记录
- CO 是分析层，从 FI 和各模块实时取数
- 模块内部联动是自动的，不需要人工在模块之间传递数据

## 六、理想态 vs 客户真实现状

> SAP 内部模块联动是真实的，但大多数企业的实际架构远比这复杂。理解这个落差，是理解 ISV 空间和产品切入点的前提。

### SAP 的理想态

SAP 的设计愿景是成为企业的统一数据底座：所有业务动作在一个系统里记录，数据自动联动，无需人工传递，月底财务自动关账，管理报表实时可查。

这个愿景在系统内部的模块联动层面是真实实现的——SD 开了销售订单，MM 自动扣库存，FI 自动生成应收，这条链在 SAP 内部确实是自动的。

### 客户的真实现状

然而，SAP 只是企业 IT 架构的一部分，而不是全部。现实中几乎所有大中型企业都运行着一个"SAP + 外部系统 + Excel"的混合架构。

**为什么会形成混合架构？**

这个问题没有单一答案，通常是以下几个因素叠加的结果：

第一，部分是战略聚焦的选择。SAP 有意将某些非核心场景交给专业工具处理，例如前端销售交给 Salesforce，因为专业 CRM 在销售流程管理上有更深的积累。这是主动划定边界，而非能力不足。

第二，部分是产品能力和体验的现实局限。SAP 的原生报表、HR 人才管理模块、用户界面，与专业工具相比确实存在差距。企业选择外部工具，往往是因为 SAP 在这些场景的体验不够好，而不完全是因为 SAP"主动放弃"。

第三，部分是历史积累的惯性。收购来的子公司跑着不同的系统，某个工厂运行着二十年前的本地软件，这些系统短期内无法替换，只能通过手动或半自动方式与 SAP 同步数据。

第四，部分是 SAP 变更周期与业务需求速度的错配。SAP 配置变更周期长（一个新需求可能需要数月才能上线），业务部门等不起，于是自己用 Excel 或 SaaS 工具解决，形成"影子系统"。

**典型的混合架构**

```
客户数据     → Salesforce（CRM，System of Record）
              ↓ 手动或集成同步
财务数据     → SAP FI/CO（ERP，System of Record）
              ↓ 数据导出
管理报表     → Power BI / Excel（分析层，非 System of Record）
HR 数据      → SuccessFactors / Personio（HR，System of Record）
              ↓ 薪资同步
薪资记录     → SAP HCM / DATEV（薪资执行层）
流程分析     → Celonis（读取 SAP 事件日志，不写入 SAP）
```

### 这个落差意味着什么

每一个系统之间的断点都是潜在的产品机会：数据不同步、口径不一致、人工操作、等待周期长。ISV 和 SaaS 产品的生存空间，大多数来自这些断点，而不是来自替换 SAP 本身。

## 七、SAP 的市场地位与壁垒

SAP 的市场地位由五个因素叠加形成。

**先发优势**
1970 年代切入市场，1980 年代拿下德国制造业核心客户，1992 年 R/3 开启全球扩张。竞争对手进入时，SAP 已经在客户系统里稳定运行十年以上。

**四层客户锁定**
- 数据锁定：几十年历史数据在 SAP 里，迁移出去的技术成本和业务风险极高
- 流程锁定：业务流程已围绕 SAP 重新设计，更换系统意味着同步重新设计所有流程
- 人员锁定：内部团队技能绑定在 SAP 上，更换系统需要大规模重新培训
- 生态锁定：周围的 SI 和 ISV 都围绕 SAP 建立，替换核心 ERP 等于同时替换整个技术生态

**德国制造业背书**
大众、西门子、拜耳、巴斯夫是 SAP 最早的标杆客户。德国顶级制造企业的使用案例是全球制造业选型时最重要的信任来源之一。

**合规深度壁垒**
SAP 在全球几十个国家做了本地化合规配置：德国劳动法、增值税申报、各国财务准则都内嵌在系统里。这个本地化积累需要几十年，不是短期可以复制的。

**竞品的结构性局限**
Oracle 技术架构更现代但德国市场本土化深度不足；Microsoft Dynamics 在复杂大型制造业场景覆盖不足；新兴云 ERP 在中型企业有竞争力但难以服务超大型复杂企业。目前没有竞品能同时满足"功能深度、本地化深度、生态规模"三个条件。

## 八、SAP 生态地图

围绕 SAP 存在六类主要角色。SAP 本身的 License 收入只是整个生态总价值的一部分，围绕 SAP 运转的服务和软件市场规模远超 SAP 本身。

**SAP 本身**
产品研发、License 销售、官方培训认证体系、SAP BTP 集成平台、SAP Store 应用市场。

**系统集成商（SI）**
Accenture、Deloitte、Capgemini、IBM、T-Systems（德国电信旗下）、Infosys、Wipro、TCS 等。负责 SAP 的实施、迁移、定制开发和长期维护运营。全球 SAP 顾问服务市场每年规模超过 300 亿美元，SI 赚的钱远超 SAP 本身的 License 收入。大部分企业通过 SI 与 SAP 打交道，而非直接与 SAP 公司合作。

**ISV 独立软件商**
在 SAP 上面或周边构建专业产品的软件公司。SAP 作为通用平台在某些专业场景能力有限，ISV 填补这些空白。代表公司：Celonis（流程挖掘）、Signavio（流程建模，已被 SAP 收购）、Informatica（数据集成）。详见第九章。

**硬件和云基础设施厂商**
AWS、Azure、Google Cloud 提供 SAP 的云托管基础设施。SAP 认证的硬件厂商包括 HP、Dell、Lenovo。

**第三方维护服务商**
Rimini Street、Spinnaker Support 等。提供替代 SAP 官方维护服务的第三方支持，价格通常显著低于官方维护费，面向希望控制长期运营成本的企业客户。

**培训和认证机构**
SAP 官方培训体系、各类 SAP 认证考试、第三方培训平台。全球持证 SAP 顾问数量达几十万，印度单独就有超过 35 万 SAP 专业人员服务全球市场。

## 九、SAP ISV 生态深入

### ISV 是什么

Independent Software Vendor，独立软件商。在 SAP 生态里特指产品依赖 SAP 数据或平台运行、但不替换 SAP 本身的软件公司。定位是在 SAP 能力薄弱的专业场景提供更深度的解决方案。

### ISV 生态存在的原因

第六章描述的混合架构中每一个断点、每一个 SAP 做不够好的场景，都是 ISV 的潜在空间。SAP 主动建设这个生态：提供 API 接口和 BTP 平台供第三方接入，开设 SAP Store 让 ISV 上架。丰富的 ISV 生态反过来也提高了客户的整体迁移成本，对 SAP 本身有利。

**ISV 与 BTP 的边界**
BTP 是 SAP 官方的扩展和集成平台，提供基础能力（API 管理、低代码开发、数据集成）。ISV 通常基于 BTP 构建，或通过 BTP 的 API 接入 SAP 数据。两者是"平台 vs 应用"的关系，不是竞争关系。但 SAP 在持续扩展 BTP 的原生能力，部分 ISV 的功能会随时间被 BTP 内化，这是 ISV 的长期风险之一。

### 主要 ISV 类别

**流程挖掘和分析**
Celonis：从 SAP 系统事件日志中分析业务流程的实际运行状况，识别效率瓶颈。市值超过 100 亿美元，是 SAP 生态中最成功的独立 ISV 之一。
Signavio：流程建模与流程挖掘工具，已被 SAP 收购并整合进 SAP 产品体系，是 ISV 被收购路径的典型案例。

**数据质量和集成**
Informatica：企业级数据治理和 ETL，负责 SAP 数据与其他系统数据的清洗、对齐和同步。
MuleSoft：API 集成平台，已被 Salesforce 收购，广泛用于 SAP 与外部系统的连接。

**用户界面层**
Neptune Software：在 SAP 底层数据基础上构建现代化移动界面，员工通过现代 App 操作，后端仍是 SAP 数据。

**自动化和数据录入**
Winshuttle：SAP 数据批量录入自动化，减少人工重复操作。
Redwood Software：SAP 作业调度与流程自动化。

**报表和 BI**
Power BI、Tableau：通过数据连接器读取 SAP 数据，提供更灵活的可视化分析能力，补充 SAP 原生报表的不足。

**合规和审计**
Oversight Systems：SAP 财务数据异常检测，支持内部审计和反欺诈。
Greenlight：SAP 系统权限管理和合规审计。

### ISV 的商业模式

**直接销售**：License 费加年度维护费，直接面向 SAP 客户销售。客单价较高，销售周期较长。

**SI 渠道合作**：与 Accenture、Deloitte 等 SI 建立合作，由 SI 在 SAP 实施项目中推荐。覆盖面广，但依赖合作关系，客户关系主导权在 SI 一侧。

**SAP Store**：在 SAP 官方应用市场上架，获得渠道背书，但需满足准入要求，SAP 收取佣金。

### ISV 面临的主要风险

**被 SAP 收购或功能内化**：做得足够重要的 ISV 可能被 SAP 直接收购（Signavio 是典型案例），或其核心功能被 BTP 或新模块原生实现。

**S/4HANA 迁移带来的适配成本**：依赖 ECC 数据结构的 ISV 产品，在客户迁移到 S/4HANA 后需要大幅重写和适配。

## 十、典型角色：痛点 × 预算 × 购买路径

> 目标：在拜访客户时快速判断——这个人有没有预算、他的真实痛点是什么、谁会最终拍板、谁会否决。

### CFO / 财务总监

**核心关注**：月底能否准时关账、财务数据准确性、审计合规、迁移投资回报。

**真实痛点**：SAP 财务数据与 CRM 销售数据口径不一致；跨系统管理报表需要人工合并；迁移的时间窗口与预算压力并存。

**预算归属**：直接控制财务系统相关预算，对影响财务合规和关账效率的工具有直接购买决策权。大型项目是重要批准人，但不一定是发起人。

**购买路径**：CFO 不会主动寻找工具，通常由 Controller 或财务 IT 发现问题、提出方案，CFO 做最终批准。能打动 CFO 的是量化的业务影响：关账时间从 T+5 缩短到 T+2，审计差异率降低多少。

**你的切入角度**：从"数据口径一致性"或"关账周期缩短"切入，用具体数字说话，避免讲技术细节。

---

### Controller 财务控制

**核心关注**：各部门成本是否在预算内、利润中心绩效、管理报告质量。

**真实痛点**：SAP 原生报表能力弱，大量工作依赖 Excel 手工处理；数据不够实时，月中看不到当前成本状态。

**预算归属**：通常没有独立的大额预算，是 BI 工具、报表工具、财务分析工具的主要需求发起人，向上汇报争取预算。

**购买路径**：典型的"影响者"而非"决策者"。愿意主动寻找工具、参与演示评估，但最终需要 CFO 或 CIO 批准。

**你的切入角度**：从"减少 Excel 手工操作时间"或"实时成本可见性"切入。Controller 对效率提升感受最直接，是最容易被打动、也最愿意内部推动采购的角色。

---

### 采购经理（Einkäufer）

**核心关注**：供应商管理效率、采购价格控制、交货准时率、合规性。

**真实痛点**：SAP MM 操作界面复杂；供应商数据散落在 SAP 和 Excel 两处；三单匹配异常需人工处理；Ariba 和 SAP 之间数据同步有延迟。

**预算归属**：通常没有 IT 工具的独立预算，需通过 CPO 或 CFO 申请。CPO 是采购流程优化类工具的主要预算控制人。

**购买路径**：采购经理是需求发起人，CPO 是决策人。涉及供应商外部协作的工具还需要 CIO 评估技术集成。

**你的切入角度**：从"减少三单匹配异常处理时间"或"供应商自助门户"切入。采购场景的流程自动化 ROI 通常比较直接，容易量化。

---

### IT 负责人 / CIO

**核心关注**：系统稳定性、S/4HANA 迁移规划与执行、集成架构、整体 IT 成本控制。

**真实痛点**：迁移成本高、周期长、业务部门配合程度不足；与外部 SaaS 工具的集成数量持续增加，维护复杂度上升；有经验的 SAP 顾问资源稀缺；业务部门绕过 IT 直接采购 SaaS 工具（Shadow IT），造成数据治理问题。

**预算归属**：IT 预算的主要控制人，对基础设施、集成平台、安全合规类工具有直接决策权。大额项目需要 CIO + CFO 联合批准。

**购买路径**：技术选型的最终决策人。关注集成可行性、安全合规、长期维护成本，不容易被"演示好看"打动，需要技术架构文档和参考客户案例。

**Veto 风险**：CIO 是最常见的否决角色。即使业务部门已认可，CIO 仍可以"集成复杂度高"、"安全合规未达标"为由否决采购。提前让 CIO 团队参与评估是规避风险的关键。

**你的切入角度**：从"降低集成维护成本"或"加速迁移"切入。需要准备具体的集成方案和 TCO 对比。

---

### 业务部门负责人（VP Sales / VP Operations 等）

**核心关注**：团队效率、业务目标达成、减少系统使用摩擦。

**真实痛点**：SAP 响应业务需求速度太慢；前端工具与 SAP 数据不同步；报表不及时，决策依赖滞后数据。

**预算归属**：通常有部门运营预算，可直接采购中小型 SaaS 工具（年费 5-20 万欧以下），不需要 IT 审批。这是 Shadow IT 的主要来源，也是中小型 SaaS 产品绕过 IT 直接切入的路径。

**购买路径**：中小型工具采购的直接决策人，大型项目需要联合 CIO 和 CFO。关注业务结果，不关心技术架构。

**你的切入角度**：从业务结果切入。"你的销售团队每周花多少时间在 SAP 和 Salesforce 之间手动同步数据？"比"我们支持 SAP API 集成"有效得多。

## 十一、SAP 项目生命周期与切入时机

> 同一个客户，在项目的不同阶段，主导者不同、预算状态不同、愿意评估的工具类型也完全不同。这一章帮你判断什么时候进、卖给谁、卖什么价值。

### 六个阶段概览

```
实施前评估 → 蓝图设计 → 构建与配置 → UAT测试 → Go-Live → 运行优化
   6-12月      3-6月       6-18月       2-4月     1-2周    持续进行
```

---

### 阶段一：实施前评估（Pre-Implementation）

**时间**：通常 6-12 个月，在正式项目启动之前。

**发生什么**：企业决定上 SAP 或迁移到 S/4HANA，开始评估方案、选择 SI、制定预算、梳理现有流程。CIO 和 CFO 是主导者，SI 开始介入做售前评估。

**预算状态**：预算正在审批中，尚未完全锁定，但方向已经确定。这是影响预算分配的最佳窗口。

**谁主导**：CIO、CFO、外部 SI 的售前顾问。

**适合卖什么**：
- 流程挖掘工具（如 Celonis）：帮助客户在实施前摸清现有流程的真实状态，识别哪些流程值得优化，避免"把烂流程原样搬进 SAP"的问题
- 数据质量评估工具：评估现有数据质量，提前发现迁移风险
- 项目管理和变更管理工具

**切入角度**："在正式实施之前，你们是否已经清楚地了解现有流程中哪些环节效率最低？我们可以帮你在 3 周内给出一份基于数据的流程诊断报告。"

---

### 阶段二：蓝图设计（Blueprint / Design）

**时间**：3-6 个月。

**发生什么**：SI 顾问与客户业务团队密集工作坊，把业务需求转化为 SAP 的配置方案。决定哪些流程用标准流程、哪些需要定制开发、哪些模块上线。这是项目中决策密度最高的阶段。

**预算状态**：预算已批，正在分配到各工作流。此时提出新的工具需求，如果能被纳入蓝图，就能获得正式预算。

**谁主导**：SI 的方案顾问（Solution Architect）、客户的项目经理和业务流程负责人。

**适合卖什么**：
- 界面层工具：如果客户在蓝图阶段决定用 Fiori 或自定义界面，现在是确定 UI 工具的时机
- 集成规划工具：帮助规划 SAP 与外部系统（Salesforce、Workday 等）的集成架构
- 培训内容工具：提前规划用户培训方案

**切入角度**：此阶段进入的关键是与 SI 建立合作关系。SI 的方案顾问在蓝图阶段有极大的工具推荐影响力，如果他们认可你的产品，会主动在蓝图文档中写入。

---

### 阶段三：构建与配置（Build & Configuration）

**时间**：6-18 个月，是整个项目中最长的阶段。

**发生什么**：根据蓝图方案，SI 团队（大量印度离岸团队参与）进行 SAP 配置、ABAP 定制开发、数据迁移脚本编写、与外部系统集成开发。

**预算状态**：预算已锁定，主要用于 SI 人力成本。此阶段引入新工具难度较大，除非能直接替代 SI 某部分工作量，降低成本。

**谁主导**：SI 的交付团队（项目经理 + 技术顾问 + 开发团队）。

**适合卖什么**：
- 测试自动化工具：减少手动回归测试的工作量，在配置变更后快速验证系统
- 数据迁移工具：加速数据清洗和迁移脚本的开发
- 开发加速工具：帮助 ABAP 开发者提高效率（代码审查、文档生成等）

**切入角度**：从"降低 SI 工时成本"切入，对 CIO 和项目经理有吸引力，因为 SI 费用是这个阶段最大的成本项。

---

### 阶段四：UAT 测试（User Acceptance Testing）

**时间**：2-4 个月。

**发生什么**：业务用户（不是 IT 团队）测试系统配置是否符合实际业务需求，发现问题提交缺陷，SI 修复后重新测试。这个阶段压力极大，项目经常因为 UAT 发现大量问题而延期。

**预算状态**：预算已接近尾声，但 UAT 发现的问题会带来额外的 SI 费用，项目经理对任何能减少缺陷数量的工具非常敏感。

**谁主导**：客户的业务团队（关键用户）+ 项目经理。

**适合卖什么**：
- 测试管理工具：帮助业务用户系统化记录和追踪测试用例、缺陷
- 培训辅助工具：帮助关键用户快速理解新系统操作，减少因"不知道怎么用"导致的误报缺陷
- 文档生成工具：自动生成操作手册和培训材料

**切入角度**："UAT 阶段发现的缺陷，平均每个需要 X 天修复，会直接推迟 Go-Live 时间。我们的工具可以帮助你减少 30% 的有效缺陷数量。"

---

### 阶段五：Go-Live 与 Hypercare

**时间**：Go-Live 本身是 1-2 周，Hypercare（高度保障期）通常持续 1-3 个月。

**发生什么**：系统正式切换上线（通常选周末做 Cutover），所有业务从旧系统迁移到 SAP。Hypercare 期间 SI 团队驻场或在线支持，快速响应生产环境中出现的问题。这是整个项目压力最集中的时刻。

**预算状态**：原始项目预算接近用完，但 Hypercare 期间出现的问题会触发额外支持费用。

**谁主导**：SI 的交付团队 + 客户的 IT 运维团队 + 业务关键用户。

**适合卖什么**：
- 实时监控工具：监控 Go-Live 后系统性能和业务流程异常
- 用户支持工具：帮助业务用户快速找到操作指引，减少对 IT 支持台的呼叫量
- 问题追踪工具：系统化管理 Go-Live 后的问题列表

**注意**：这个阶段不适合引入新的复杂工具，客户的注意力和精力全部集中在稳定系统上。最适合的产品是轻量、快速部署、能立即减少压力的工具。

---

### 阶段六：运行优化期（Run & Optimize）

**时间**：Go-Live 之后，持续进行，没有终点。

**发生什么**：系统稳定运行后，企业开始关注如何从 SAP 获取更多价值——优化流程、提升用户效率、扩展到新的模块或业务单元、分析数据找到改进机会。这个阶段与实施项目不同，没有明确的终止点，需求持续产生。

**预算状态**：从项目预算转变为运营预算（OpEx）。运营预算通常按年度审批，更稳定，但单笔金额比项目预算小。订阅制 SaaS 产品在这个阶段最容易被接受。

**谁主导**：业务流程负责人、COE（卓越中心，Center of Excellence）团队、IT 运维团队。COE 是很多大型 SAP 客户在 Go-Live 之后设立的内部团队，负责 SAP 的持续优化和新需求管理。

**适合卖什么**：
- 流程挖掘和分析（Celonis 类）：识别流程中的瓶颈和浪费，持续优化
- BI 和报表工具：帮助业务团队从 SAP 数据中获取洞察
- 用户培训和采用率工具：提升 SAP 使用率，减少"有功能但没人用"的情况
- 自动化工具：把高频手工操作自动化，如 RPA、Power Automate
- 模块扩展：在核心模块稳定后，扩展到更多业务域（如 Ariba 采购协作、SuccessFactors HR）

**切入角度**：这是最适合产品主导增长（PLG）策略的阶段。客户已经了解 SAP，已经有了明确的业务痛点，决策周期比实施阶段短，愿意为解决具体问题付费。

---

### 项目阶段与切入时机汇总

| 阶段 | 主导者 | 预算状态 | 最适合的工具类型 | 进入难度 |
|------|--------|---------|----------------|---------|
| 实施前评估 | CIO、CFO、SI 售前 | 审批中，可影响 | 流程诊断、数据质量评估 | 低，窗口期开放 |
| 蓝图设计 | SI 方案顾问、业务负责人 | 已批，分配中 | 界面工具、集成规划 | 中，需通过 SI |
| 构建配置 | SI 交付团队 | 已锁定 | 测试自动化、数据迁移 | 高，需替代 SI 工作量 |
| UAT 测试 | 业务关键用户、项目经理 | 接近尾声 | 测试管理、培训辅助 | 中，痛点明确 |
| Go-Live/Hypercare | SI + IT 运维 | 接近用完 | 监控、轻量支持工具 | 高，注意力不在采购 |
| 运行优化 | COE、业务负责人 | 运营预算，稳定 | 流程挖掘、BI、自动化 | 低，需求持续产生 |

**最佳切入窗口**：实施前评估期（影响预算分配）和运行优化期（需求持续、决策周期短）。S/4HANA 迁移压力使得大量企业目前处于"实施前评估"状态，这是当前市场最大的窗口。

## 十二、SAP 切入点判断框架

> 用五个维度帮你判断一个产品机会是否值得进入，以及如何进入。

### 五个判断维度

**① 高频痛点**
这个痛点是每天/每周都在发生，还是偶尔才遇到？

高频痛点的工具采购优先级高，ROI 容易量化。例：财务人员每天手动从 SAP 导数据到 Excel 做报表，这是高频痛点，BI 工具容易卖。

低频痛点（每季度/每年）的工具很难成为优先采购项，即使客户认可问题存在。例：年度审计合规检查，客户平时感知不到痛，到审计季才想起来。

**② 触发项目**
客户现在是否有一个正在进行的大项目，你的产品可以作为其中一部分切入？

S/4HANA 迁移是当前最大的触发项目——几乎所有正在迁移的企业都在评估配套工具（数据清洗、迁移测试、流程再造、新界面层）。在迁移项目窗口期内切入，比平时冷启动容易得多。各阶段具体的切入时机详见第十一章。

其他常见触发项目：ERP 实施（新建或扩展模块）、数字化转型项目、合并收购后的系统整合、合规审计整改。

**③ 预算归属**
这个问题的预算在谁手里？是 IT 预算还是业务部门预算？

IT 预算：采购周期长（6-18 个月），需要经过 CIO 评估和 CFO 批准，合同金额通常较大。适合有完整销售团队的成熟产品。

业务部门预算：采购周期短（1-3 个月），业务负责人可直接决策，合同金额通常较小（年费 5-20 万欧）。适合产品主导增长（PLG）策略。

**④ 集成难度**
你的产品需要与 SAP 集成到什么程度？结合第四章部署形态来判断。

只读数据（报表、分析）：集成相对简单，通过 SAP 标准 API 或 BTP 连接器即可实现，对所有部署形态均适用，CIO 阻力较小。

读写数据（双向同步）：集成复杂度高，需要考虑数据一致性、错误处理、权限控制。On-Prem 客户相对灵活，Public Cloud 客户受限较多。

替换 SAP 模块：几乎不可能直接替换，通常需要长期共存策略。

**⑤ 替代风险**
SAP 自己（通过 BTP 或新模块）能否在 1-3 年内提供类似能力？

如果你的产品做的是 SAP 明确在路线图上要覆盖的功能，长期风险较高。需要评估你的差异化是否足够深、客户切换成本是否足够高。

如果你的产品做的是 SAP 不会覆盖的场景（如深度行业垂直化、特定地区合规、前端用户体验层），替代风险较低。

### 判断矩阵

| 维度 | 高价值信号 | 低价值信号 |
|------|-----------|-----------|
| 高频痛点 | 每天/每周发生 | 每季度/年度才遇到 |
| 触发项目 | S/4HANA 迁移中 / 大型项目进行中 | 无明确项目背景 |
| 预算归属 | 业务部门可直接决策 | 纯 IT 预算，需多层审批 |
| 集成难度 | 只读数据，标准 API 可实现 | 双向写入，涉及核心财务数据 |
| 替代风险 | SAP 路线图未覆盖 | SAP BTP 正在开发类似功能 |

### 三种典型切入路径

**路径一：从断点切入**
找到客户混合架构中最痛的断点（CRM-SAP 同步、报表手工合并、流程审批延迟），提供专门解决这个断点的工具。优势是痛点明确、ROI 容易量化；劣势是单点工具容易被替代或被 SAP 内化。

**路径二：搭 S/4HANA 迁移的顺风车**
在客户的迁移项目窗口期内切入，把你的产品定位为"迁移加速器"或"迁移后的能力升级"。具体进入时机参考第十一章各阶段分析。优势是客户已经有预算和项目背景；劣势是需要与 SI 合作，且项目结束后需要证明持续价值。

**路径三：从业务部门进入，绕过 IT**
先让业务团队用起来，建立使用习惯和数据积累，再通过业务部门影响 IT 层面的正式采购。优势是起步快、决策链短；劣势是可能遭遇 CIO 的否决，需要最终补齐 IT 合规要求。

## 十三、和 SAP 打交道必须知道的事情

### 核心术语

**Transaction Code（T-Code）**
SAP 里的功能操作码，每个业务功能对应一个代码。ME21N 是创建采购订单，FB60 是录入供应商发票。听到客户提到某个 T-Code，可以快速判断他们在操作哪个业务环节。

**Client**
SAP 里的最高层级隔离单位，相当于一个独立的企业实例。大企业通常维护开发、测试、生产三个独立 Client 环境。

**Transport**
配置变更从开发环境迁移到生产环境的机制。"等 Transport"是 SAP 项目里最常见的进度等待原因，一个小改动可能需要几周才能上线。

**Go-Live**
SAP 系统正式上线的节点，是项目周期中风险最集中的时刻。

**Cutover**
Go-Live 前将旧系统数据迁移到 SAP 的过程，通常安排在周末窗口期内完成，时间紧张、操作密集。

**ABAP**
SAP 的专有编程语言，用于定制化开发。项目中涉及 ABAP 开发意味着较高的时间和成本投入，且依赖稀缺的 ABAP 开发者资源。

**System of Record**
当同一类数据在多个系统中存在时，以哪个系统为权威来源。SAP 是财务数据的 System of Record，意味着财务数字以 SAP 为最终依据，具有法律效力，不可随意删改。

**Three-way Match（三单匹配）**
采购订单、收货单、供应商发票三张单据的金额必须一致，SAP 才会放行付款。这是采购内控的核心机制，异常处理是采购团队的高频痛点。

**TCO（Total Cost of Ownership）**
总拥有成本。对 SAP 而言包括 License 费、年度维护费（通常为 License 费的 22%）、实施费、定制开发费、内部人力成本、以及迁移费用。与 CIO 和 CFO 谈判时，TCO 是关键数字。

**ECC vs S/4HANA**
ECC 是老版核心 ERP，运行在传统关系型数据库上，SAP 宣布 2030 年停止官方支持。S/4HANA 是新版，运行在 HANA 内存数据库上，处理速度更快，数据模型更简洁，支持云部署。两者功能逻辑基本相同，主要差别是技术架构和界面。

**Clean Core**
SAP 推广的一种实施理念：尽量不做定制开发，使用标准功能，把扩展放到 BTP 上而不是直接修改 SAP 核心代码。这样迁移和升级成本更低。Public Cloud 客户几乎被强制遵守，On-Prem 客户可以选择是否采用。

**COE（Center of Excellence）**
大型 SAP 客户在 Go-Live 之后通常设立的内部卓越中心团队，负责 SAP 的持续优化、新需求管理、用户培训和最佳实践推广。COE 是运行优化期最重要的内部利益相关方，也是 ISV 产品在客户内部的主要推动者。

### 必须知道的常识

**SAP 不是开箱即用的产品**
每个企业的 SAP 都需要根据自身业务流程做配置和适配，这个过程称为实施，通常需要引入外部顾问，周期从几个月到几年不等。实施费通常远超 License 费本身。

**大多数人不直接操作 SAP 界面**
直接使用 SAP 原生界面的是财务、采购、仓库、HR 等操作岗位。管理层、销售、产品负责人通常接触的是从 SAP 导出整理后的报表。他们对 SAP 的痛点感知通常表现为"数据出不来"、"流程推进慢"、"需求变更响应周期太长"。

**SI 在客户关系中扮演核心角色**
大部分企业通过系统集成商实施和维护 SAP。如果你的解决方案能够获得主要 SI 的推荐，客户触达效率会显著高于直接销售。但与 SI 合作意味着部分客户关系和定价主导权转移到 SI 一侧，需要权衡。

**德国客户对数据主权和合规格外敏感**
GDPR 合规、数据不出欧盟、本地化部署选项，是德国 CIO 在评估任何新工具时的标配问题。提前准备好数据处理协议（DPA）和合规文档，是进入德国企业采购流程的基本门槛。如果你的产品只支持云端部署，进入德国大企业的阻力会比进入美国企业大得多。
